# QA 관점 브레인스토밍 — spiceMap

> flow_map.md 기반 | 2026-04-08

## QA 전략 개요

spiceMap은 **데이터 파이프라인 + 분석 모듈 + 시각화 UI** 세 계층으로 구성된다.  
각 계층의 오류는 서로 다른 방식으로 결과물에 영향을 미치며, 경진대회 발표 시 치명적 실패를 유발할 수 있다.

```
[데이터 계층] → 수집/정렬 오류가 모든 분석에 전파
[분석 계층] → 모듈별 로직 오류가 시각화와 정책 제언에 반영
[시각화 계층] → 렌더링/인터랙션 오류가 사용자 경험을 직접 파괴
```

## 1. 데이터 품질 검증

### 1.1 8개 데이터셋 단위 정렬 검증

flow_map.md의 핵심 리스크: **시간(시간대) + 월 + 분기** 세 단위를 `상권-분기` 기준으로 통합해야 함.

| 검증 항목 | 테스트 방법 | 합격 기준 |
|---------|-----------|---------|
| OD 데이터(월) → 분기 집계 | 3개월 합산 후 샘플 비교 | 분기 합계 오차 < 1% |
| 생활인구(시간대) → 분기 대표값 | 피크 시간대 / 평균 / 합계 중 선택 일관성 | 집계 방식 문서화 + 재현 가능 |
| 매출/폐업(분기) 직접 사용 | 상권 코드 매칭 누락률 | 누락 < 5% |
| 실시간 도시데이터 → 캐시 | 장애 시 마지막 캐시 날짜 표시 | 캐시 만료 기준 명확화 |

### 1.2 결측치 처리 정책

- **허용 결측 상권 비율**: 분기당 10% 초과 시 해당 분기 분석 제외 (기준 문서화 필요)
- **GRI 임대료 항목 결측 시**: 0.25 계수 재분배 규칙 정의 필요
  - 예: 임대료 없을 경우 → 프랜차이즈 비중에 0.35 재할당? 분석 포함하되 경고 표시?
- **신규 상권 (이력 없음)**: 유형 분류 제외 또는 "데이터 부족" 별도 표시

### 1.3 이상치 탐지

| 이상치 유형 | 탐지 방법 | 처리 방침 |
|-----------|---------|---------|
| 매출 음수 값 | 단순 범위 검사 | 제거 + 로그 기록 |
| OD 유출 > 전체 인구 | 구 정규화 후 백분위 초과 검사 | 캡핑 + 경고 플래그 |
| 분기 간 매출 500% 급증 | IQR 기반 이상치 | 조사 후 포함/제외 결정 |
| 상권 코드 불일치 | 데이터셋 간 조인 키 검사 | 매핑 테이블 관리 |

## 2. 모듈별 테스트 전략

### Module A — 상권 흐름 네트워크

**핵심 로직**: 상권을 노드, 이동을 방향성 엣지로 구성한 유향 그래프  
**계산 지표**: `in_flow`, `out_flow`, `net_flow`, `centrality`

| 테스트 케이스 | 검증 방법 | 합격 기준 |
|------------|---------|---------|
| net_flow = in_flow - out_flow | 단위 테스트 | 모든 상권에서 정확 일치 |
| 유향 그래프 엣지 방향성 | 역방향 엣지 존재 여부 확인 | A→B와 B→A가 독립적으로 존재 |
| 중심성(centrality) 계산 | NetworkX 기준 결과 비교 | 알려진 고중심 상권(강남, 홍대)이 상위 | 
| 상권 유형 분류 경계 | 경계값 테스트 (net_flow = 0 근방) | 안정형과 흡수형 분류 명확 |
| 고립형_단절 탐지 | 엣지 수 0 또는 in_flow ≈ 0 | False negative < 5% |

**엣지 케이스**:
- 신규 상권 (1개 분기만 데이터) → 중심성 계산 불가 처리
- 상권 간 OD 데이터 없는 경우 → 엣지 없음으로 처리, 고립형 분류 여부 확인

---

### Module B — 젠트리피케이션 위험 지수 (GRI)

**계산식**:
```
GRI = 0.30*순유입증가 + 0.25*임대료상승 + 0.20*프랜차이즈비중증가 + 0.15*매출급증 - 0.10*독립소상공인비중
```

| 테스트 케이스 | 검증 방법 | 합격 기준 |
|------------|---------|---------|
| 계수 합산 검증 | 0.30+0.25+0.20+0.15-0.10 = 0.80 ≠ 1.0 | **⚠️ 계수 합산이 0.80 — 의도적인가? 확인 필요** |
| GRI 범위 | 모든 상권 샘플 GRI 값 범위 확인 | 0.0 ~ 1.0 정규화 여부 명확화 |
| 임대료 항목 결측 시 | 임대료 데이터 없는 상권에 GRI 계산 | 경고 표시 + 부분 GRI 레이블 |
| 독립소상공인 비중 = 0% | 극단값 처리 | GRI 계산 가능, 0으로 처리 |
| 경계값: GRI가 정확히 임계값 | 높은/낮은 위험 분류 경계 | 상위 25%가 높은 위험? 절대값 기준? |

**발견된 잠재 버그**:
> 계수: 0.30 + 0.25 + 0.20 + 0.15 - 0.10 = **0.80**  
> 합계가 1.0이 되지 않는다. 의도적 가중치 설계인지, 오탈자인지 확인 후 문서화 필요.

---

### Module C — 흐름 단절 구간 탐지

**탐지 대상**: 상권으로 유입이 오지 않는 상류 구간 (보행 접근, 교통 결절, 야간 체류, 목적 편향)

| 테스트 케이스 | 검증 방법 | 합격 기준 |
|------------|---------|---------|
| 보행 접근 단절 탐지 | 보행 데이터와 OD 간 불일치 | False positive < 10% |
| 교통 결절 탐지 | 대중교통 접근성 지표 연계 여부 | 연계 데이터셋 식별 필요 |
| 목적 편향 탐지 | 출근/쇼핑 비율 극단값 | 단일 목적 > 90%를 편향으로 정의 |
| 야간 체류 단절 | 야간 생활인구 급감 | 주간 대비 90% 감소 기준 |
| 중복 단절 원인 | 여러 원인이 동시 존재 | 우선순위 표시 로직 필요 |

**미정 기준**:
- "단절"의 정량적 임계값이 flow_map.md에 명시되지 않음 → 팀 내 기준 합의 필요
- 교통 결절 데이터셋이 8개 목록에 없음 → 추가 확보 필요 여부 확인

### Module D — AI 정책 제언

**출력 포맷**: `현황`, `근거`, `권고3개`, `담당부서`, `기대지표`

| 테스트 케이스 | 검증 방법 | 합격 기준 |
|------------|---------|---------|`
| 포맷 필드 완전성 | 5개 필드 모두 존재 여부 | 100% 완전성 |
| 권고 항목 수 | 권고가 정확히 3개인지 | 항상 3개 (초과/미달 불가) |
| 담당부서 매핑 정확도 | 상권 유형별 부서 매핑 테이블 검증 | 흡수형_과열 → 임차인보호 일치 여부 |
| 규칙 기반 동작 확인 | 동일 입력 → 동일 출력 (결정론적) | 100% 재현 가능 |
| 빈 입력 처리 | 분석 데이터 없는 상권에 제언 요청 | "데이터 부족으로 제언 불가" 출력 |
| 극단 입력 처리 | GRI = 최대값, 폐업률 = 100% | 오류 없이 처리, 최고 위험 제언 출력 |

**환각 방지 검증** (LLM 사용 시):
- 데이터 근거 없는 주장 포함 여부 → 규칙 기반 근거 강제 필드 확인
- 허구 담당 부서 출력 여부 → 허용 부서 화이트리스트 매칭

### Module E — 정책 우선순위 점수

**계산식**:
```
Priority = 0.35*순유출위험 + 0.25*폐업위험 + 0.15*연결단절 + 0.15*생활권격차 + 0.10*GRI
```

| 테스트 케이스 | 검증 방법 | 합격 기준 |
|------------|---------|---------|
| 계수 합산 | 0.35+0.25+0.15+0.15+0.10 = 1.0 | ✅ 합계 = 1.0 확인 |
| 분류 임계값 경계 | Priority = 80, 79, 60, 59 값 입력 | 80 → 즉시, 79 → 연내, 59 → 모니터링 |
| 입력 정규화 | 각 하위 지표 0~1 정규화 여부 | 합산 전 스케일 통일 확인 |
| 순위 안정성 | 동점 처리 방식 | 동점 시 정렬 기준 명확화 |
| GRI 미계산 상권 | GRI 항목 0.10 처리 방식 | 0으로 처리 또는 제외 + 표시 |

## 3. 검증 가설 (H1~H3) 테스트

### H1: 순유입 증가 → 매출 증가 (상관분석)
- **테스트**: Pearson r 계산, p-value < 0.05
- **합격 기준**: r ≥ 0.3 (약한 상관) / r ≥ 0.5 (중간, 권장)
- **실패 시 대응**: 지연 효과(Lag) 고려 → 1분기 지연 상관 추가 분석

### H2: 흐름 단절 → 폐업률 상승 (예측력 검증)
- **테스트**: 단절 탐지 상권 vs 비탐지 상권의 폐업률 비교 (t-test or Mann-Whitney)
- **합격 기준**: p < 0.05, 효과 크기(Cohen's d) ≥ 0.3
- **실패 시 대응**: 단절 정의 재검토, 시간 범위 조정

### H3: 순유입↑ + 임대료↑ + 프랜차이즈↑ → 젠트리피케이션 위험
- **테스트**: 3가지 조건 모두 충족 상권의 GRI 값 비교 (ANOVA)
- **합격 기준**: 조건 충족 그룹 GRI 평균 > 미충족 그룹 + 통계적 유의성
- **실패 시 대응**: 가중치 재조정, 임대료 대체 지표 검토

## 4. 시각화 E2E 테스트

### 3분 발표 시나리오 체크리스트

```
[ ] Step 1: 오전 8시 출근 흐름 필터 → 강남·여의도 흡수형 노드 강조 표시
[ ] Step 2: 오후 2시 쇼핑 흐름 전환 → 홍대·성수 집중 OD 곡선 변화
[ ] Step 3: 침체 상권 클릭 → 우측 패널에 단절 구간 원인 + 정책 제언 카드 표시
[ ] Step 4: 분기 타임라인 전환 → 악화/회복 추세 시각적 변화 (색상, 크기)
[ ] 전체 시나리오 재생 시간 < 3분
[ ] 모든 단계에서 로딩 지연 < 2초
```

### 인터랙션 테스트

| 인터랙션 | 테스트 항목 | 합격 기준 |
|---------|-----------|---------|
| 상권 노드 클릭 | 패널 업데이트 속도 | < 500ms |
| 시간대 슬라이더 | OD 곡선 애니메이션 매끄러움 | 60fps or 끊김 없음 |
| 분기 전환 | 노드 색상/크기 재계산 | < 1초 |
| 자치구 필터 | 해당 구만 강조 | 다른 구 opacity 감소 정확 |
| 줌인/아웃 | 노드 레이블 가독성 | 줌 레벨별 레이블 표시 기준 |

## 6. 리스크 기반 테스트 우선순위 매트릭스

```
높은 영향 + 높은 확률 (즉시 테스트)
├── GRI 계수 합산 이슈 (0.80 vs 1.0)
├── OD/매출/생활인구 단위 정렬 오류
└── 3분 발표 시나리오 전체 흐름 E2E

높은 영향 + 낮은 확률 (사전 준비)
├── 실시간 데이터 장애 시 fallback
├── AI 제언 환각 방지 (LLM 사용 시)
└── H1~H3 가설 검증 실패 시 대응 플랜

낮은 영향 + 높은 확률 (반복 자동화)
├── 데이터셋 결측치 처리 로그
├── 이상치 탐지 자동화
└── 포맷 필드 완전성 검사

낮은 영향 + 낮은 확률 (모니터링)
├── 동점 처리 엣지 케이스
└── 신규 상권 (이력 없음) 처리
```

## 7. QA 핵심 발견 사항 (flow_map.md 검토 결과)

1. **GRI 계수 합계 = 0.80**: `0.30+0.25+0.20+0.15-0.10 = 0.80` → 정규화 공식 or 오류 여부 확인 필수
2. **"단절"의 정량 임계값 미정**: Module C에서 단절을 탐지하는 수치 기준이 없음 → QA 기준 설정 불가
3. **교통 결절 데이터 출처 불명확**: 8개 데이터셋에 교통 접근성 데이터 없음 → 추가 확보 또는 분석 제외 결정 필요
4. **H1~H3 검증 실패 시 대응 프로토콜 없음**: 가설이 기각될 경우 모델을 어떻게 수정할지 미정
5. **Module D 규칙 정의 문서 부재**: AI 제언 규칙이 코드에만 있다면 QA 불가 → 별도 규칙 문서화 필요
