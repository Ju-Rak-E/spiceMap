## 1. 프로젝트 목표 & 성공 지표

### 목표
서울 상권 불균형(과열 + 침체 동시 문제)을 **데이터 기반으로 시각화**하고, 정책 담당자가 실제로 활용 가능한 인터랙티브 플랫폼을 만든다.  
**일차 목표**: 2026 서울시 빅데이터 활용 경진대회 시각화 부문 수상.  
**이차 목표**: 서울시 정책 현장에서 실제 채택되는 도구.

### 정량 KPI

| KPI | MVP 기준 | 완성본 기준 |
|-----|---------|-----------|
| 분석 상권 커버리지 | 2개 자치구 | 서울 전역 (424개 상권) |
| GRI 예측 정확도 | 상관계수 r ≥ 0.5 (H1) | r ≥ 0.7 |
| 정책 제언 포맷 준수율 | 100% (현황/근거/권고3/부서/지표) | 100% |
| 3분 발표 시나리오 재현 | 4개 스텝 모두 동작 | 전체 필터 동작 |
| 실시간 데이터 장애 시 fallback | 캐시 재생 가능 | 자동 전환 |

### 미정 KPI (결정 필요)
- [ ] 경진대회 심사 배점 기준 확인 필요 (시각성? 정책 실효성? 데이터 품질?)
- [ ] 서울시 실제 채택 시 성과 측정 주기 (분기? 연간?)

## 2. 범위 결정 — MVP vs 완성본

### MVP (1~2주) — 확정된 기능
| 기능 | 포함 여부 | 비고 |
|------|---------|------|
| 자치구 1~2곳 시범 | ✅ | 어느 자치구? → **결정 필요** |
| 출근/쇼핑 2가지 목적 흐름 | ✅ | |
| 상권 유형 분류 (5종) | ✅ | 흡수형_과열, 흡수형_성장, 방출형_침체, 고립형_단절, 안정형 |
| GRI 1차 버전 | ✅ | 계수 고정, 임대료 데이터 없으면 0.25 항목 임시 제외? |
| 정책 제언 템플릿 기반 | ✅ | AI 없이 규칙 기반 |

### 완성본 (7주) — 위험 기능
| 기능 | 리스크 | 대응 방안 |
|------|--------|---------|
| 서울 전역 확장 | 데이터 처리 성능 | 상권-분기 단위 사전 집계 |
| 시간대/요일 전체 필터 | UI 복잡도 폭증 | 필터 우선순위 사전 결정 |
| AI 스트리밍 설명 | 환각 위험, 응답 지연 | 고정 포맷 + 근거 강제 규칙 |
| 자동 리포트 생성 | 포맷 표준화 어려움 | PDF 템플릿 고정 |

### 범위 크리프 위험 신호
- 분석 모듈이 5개 → 각 모듈이 독립적으로 복잡해질 수 있음
- D3.js + MapLibre + Deck.gl 3가지 시각화 라이브러리 동시 사용 → 기술 통합 비용
- AI 제언(Module D)이 가장 마지막인데 발표에서 핵심 포인트가 될 경우 우선순위 충돌

## 3. 이해관계자 맵

```
[데이터 운영팀]
    ↓ 월간 적재, 상권 흐름 요약표
[서울시 정책팀]
    ↓ 분기 분석, 위험 상권 목록
[자치구 경제과]
    ↓ 현장 검토, 자치구 의견
[본청 + 자치구] → 지원 결정 (활성화/보호/접근성)
[정책평가팀] ← 사후 평가 피드백
[경진대회 심사위원] ← 3분 발표 직접 평가
```

### 이해관계자별 핵심 니즈

| 이해관계자 | 원하는 것 | 우려하는 것 |
|-----------|---------|-----------|
| 서울시 정책팀 | 실행 가능한 우선순위 목록 | 데이터 편향, 현장 괴리 |
| 자치구 경제과 | 우리 구만 필터링, 현장 맥락 반영 | 수치가 현실과 다를 경우 |
| 경진대회 심사위원 | 임팩트 있는 시각화 + 논리적 근거 | 과장된 AI 활용 주장 |
| 데이터 운영팀 | 자동화된 파이프라인 | 수동 개입 최소화 |

## 4. 리스크 관리

### 리스크 매트릭스

| 리스크 | 발생 확률 | 영향도 | 대응 전략 |
|--------|--------|--------|---------|
| OD ≠ 실제 소비 불일치 | 높음 | 높음 | 상관 검증(H1~H3) 선행, 한계 명시 |
| 임대료 데이터 취득 불가 | 중간 | 높음 | GRI 계수 재조정 또는 대체 지표 |
| 7주 일정 내 서울 전역 데이터 처리 지연 | 높음 | 중간 | MVP를 발표 기준으로 확정, 전역은 데모 |
| 실시간 도시데이터 API 장애 | 중간 | 낮음 | 캐시 + 재생 모드 구현 |
| 심사위원이 AI 환각 문제 제기 | 낮음 | 높음 | 규칙 기반 명시, "생성형 AI 아님" 표현 |
| 자치구별 데이터 규모 편차 | 높음 | 중간 | 구 정규화 + 백분위 변환 |

## 5. 개발 로드맵 검토

### 7주 마일스톤 & 위험 포인트

```
Week 1-2: 데이터 수집 + 공간 결합 + 시계열 정렬
  ⚠️ 위험: OD 데이터(월) + 매출(분기) + 생활인구(시간) 단위 정렬 복잡도
  → 공통 기준: 상권-분기 단위 확정 필요 (Week 1 말 검증 체크포인트 추가)

Week 3-4: 흐름 분석 모듈 + KPI 검증
  ⚠️ 위험: H1~H3 가설 검증 결과가 기대와 다를 경우 모델 수정 비용
  → 검증 실패 시 범위 조정 프로토콜 사전 정의 필요

Week 5-6: UI/지도/패널 구현
  ⚠️ 위험: D3.js + MapLibre + Deck.gl 통합 기술 난이도
  → 라이브러리 선택 재검토: 하나를 기준으로 나머지 보완 역할로 제한

Week 7: AI 연동 + 발표 준비
  ⚠️ 위험: AI 연동이 마지막 주 → 문제 발생 시 발표에 영향
  → Week 6 말에 AI 없는 버전(완성)을 발표 가능 상태로 확보
```

## 6. 미결 의사결정 목록

- [ ] **MVP 대상 자치구 결정**: 성수(과열)+ 침체 상권 1개? 데이터 취득이 쉬운 구?
- [ ] **GRI 임대료 데이터 확보 가능 여부**: 없을 경우 대체 지표 or 계수 재분배
- [ ] **경진대회 발표 포맷**: 3분 시나리오 단일? 질의응답 포함 여부?
- [ ] **Module D AI 제언 모델**: 외부 LLM API 호출? 완전 규칙 기반? 비용 예산?
- [ ] **정책 우선순위 점수 80+ 기준**: 임계값 근거가 무엇인가? 전문가 검토 필요?

## 7. PM 관점 핵심 질문

2. **"3분 발표에서 심사위원을 설득할 핵심 메시지 하나는?"** — "흐름이 왜 기울어졌는가"를 1분 안에 보여줄 수 있는가?
3. **"AI가 없어도 가치 있는가?"** — Module D를 제거해도 나머지 4개 모듈만으로 완결성이 있는가?
